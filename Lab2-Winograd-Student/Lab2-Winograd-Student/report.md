# Lab 2 report

## Q1: Strassen

测量朴素矩阵乘法和 Strassen 的平均运行时间。实际结果如下：

| Matrix Size | Naive Time (s) | Strassen Time (s) | Speedup |
|---|---:|---:|---:|
| 256 x 256 | 0.024407 | 0.00938497 | 2.60065x |
| 512 x 512 | 0.244691 | 0.063119 | 3.87666x |
| 1024 x 1024 | 5.18433 | 0.435526 | 11.9036x |

### 分析

从表中可以看到，Strassen 在三个测试规模上都比朴素矩阵乘法更快，而且随着矩阵规模增大，速度优势逐渐扩大，原因主要有以下几点；

1. 乘法次数减少的收益在大规模时更明显：Strassen 每一层把 8 次乘法减少为 7 次。随着递归深度增加，累计节省的乘法数量迅速放大，因此在 `1024 x 1024` 这类大矩阵上收益最明显。

2. 递归开销在小规模问题中占比更高:Strassen 需要额外的矩阵加减、子块复制、递归调用、空间分配。因此在小规模矩阵下，这些开销会抵消部分收益。故 `256 x 256` 矩阵乘法加速比明显低于 `1024 x 1024`。

3. base case 对性能影响很大:本实现当 `n <= 64` 时使用朴素乘法而不是继续递归，减少了小矩阵下的的函数调用和内存管理成本。若阈值设置过小，递归层数增加，常数开销会更高；若阈值设置过大会丢失 Strassen 的乘法优势，阈值决定了实际的 crossover point。

4. cache 也会影响真实性能:Strassen 把大矩阵拆成子块后，子问题规模逐步缩小，更容易增大cache 命中率。当前实现采用了较直接的子块复制与重新分配方式，没有针对 cache 做特别优化。

## Q2: Winograd

Winograd 测试参数为：

- `C_in = 3`
- `C_out = 64`
- `H = 56`
- `W = 56`
- `K = 3`
- `stride = 1`
- `pad = 0`

实际测得运行结果如下：

| Method | Time (s) | Relative Speed |
|---|---:|---:|
| Direct Convolution | 0.000538278 | 1.00x |
| Winograd F(2x2, 3x3) | 0.0028877 | 0.186403x |

这里 `Relative Speed` 以直接卷积为基准，因此当前实现里 Winograd 并没有加速，反而更慢。

### 分析

Winograd 理论上减少了乘法次数，但当前测试结果显示其性能明显落后于直接卷积，原因主要体现在以下几个方面。

1. FLOPs reduction 只减少了乘法，不减少全部成本：Winograd 的优势主要来自将每个 `2x2` 输出块所需的乘法数量从 36 次降低到 16 次。但实际程序运行时间不仅由乘法决定，还包含输入变换、核变换、输出逆变换、循环控制和内存读写，这些操作带来额外成本。

2. ransform overhead ：在当前实现中：每个卷积核都需要先做 `G g G^T` 变换，每个输入 tile 都要做一次 `B^T d B`，再做 `A^T M A` 逆变换。这些变换涉及矩阵运算，产生常数开销，对本实验规模不大的输入而言，transform overhead 占比很高，导致 Winograd 变慢。

3. 缓存不够友好：Winograd 的执行流程包含对输入进行 tile 划分，为每个 tile 构造局部 `4x4` 数组，对中间矩阵 `U`、`V`、`M` 做频繁访问，在不同阶段切换数据布局。访问模式相比直接卷积更复杂。当前实验没有做专门的优化，产生额外的内存访问成本。

4. baseline 本身很小，实验直接卷积的输入规模为 `56 x 56 x 3`，输出通道为 64，batch 为 1。这个规模下，直接卷积的三重循环实测时间只有 `5.38e-4` 秒。Winograd 减少乘法数量相较于额外的常数开销不能带来明显受益。
